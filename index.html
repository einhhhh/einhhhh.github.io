关于'孤独'

发表于 2020-01-20 Valine：

回想之前跨年的时候，人嘛，每每到一个自以为有所意义的节点便开始感慨。成群成对的人怀着或是满足或是幽怨或是快乐或是伤感的心情写下点什么，po上社交媒体，希望得以引发共鸣。但真正如此被引起了共鸣的人又有几何呢？我始终认为，没有一个人能够真正走进另一个人的内心，能够在他人身上感受到什么，我自认为如此即是真理，而事实上如此确实真理。往往人们分享、人们表达，然后遭遇冷遇、被泼冷水，而后借鲁迅先生那句被用烂的的句子感慨“人与人的悲欢是不相同的”。人与人的悲欢何时相通过？人呐，因为那刻进基因里的对他人认同的渴望，总是抱有期望，抱有幻想，想象着那悲欢相同的人与人。
他们应该意识到，孤独才是人与人之间的本质。
心理学上有一个理论——我是个不爱记的人，所以具体的理论条目是何我已记不太清，相关的点是在罗素米勒的《亲密关系》里看到的——人与人之间的交往只不过是价值之间的交换罢了。我认为很合理，和我之前自己的推论是较为拟合。所以嘛，人与人之间的关系需要经营——经营以促进价值交换嘛，朋友间要多聚聚友情才不会冷淡，夫妻间要多互动多恩爱恩爱才能让第三者无处插缝。因此，当某天朋友之间疏远了，是你我间没有值得交换的价值值得经营了；当某天婚姻破裂了，是夫妻间没有有价值的东西能得以让经营继续了。是啊，经营而已，感情、维系不过是经营的产物罢了，根本上只为了让最多的好处流向了自己。
这样推下来一个很悲伤的很悲观的一点是：人与人之间就是如此看似荒诞实则如此，所谓联系所谓交际，不是心灵共鸣，不是思想交融，不是灵魂碰撞，只是满足个人归属感，对他人认可渴望感的工具罢了。
无论怎样下来，我们都是孤独的。
但我想表达的不止是这些东西，我想说的是在这些悲观的让人绝望的观点后面的能稍微带来点光明的东西。
我曾经有一段时间一直在找寻一些东西，找寻的东西说实话蛮可笑的。我在找寻生活的意义。我的逻辑点是：“既然人终将成那一抔黄土，人类很大概率终将泯于宇宙，那我的存在有何意义，人类的存在有何意义？”我的一个朋友笑话我：“你这是空心病，别想太多了。”但我并不认为如此，我认为这是人生一大事，我不想浑浑噩噩过完这一生，我若连我存在为何都没弄清楚，那我如何而活，如何好好活？
我找寻意义，但我终究发现毫无意义。我曾相信存在主义能给我带来光明，但终究觉得自己虚伪逃避。最后当我选择直面这无尽虚无时，顿感豁然开朗：既然无意义，那就无意义，与其郁郁寡欢，不如欢乐过完此生。
我就是我想表达的光明点的东西：既然本质孤独，那就孤独，与其为此悲伤，不如与人和睦快乐过完这生。其实这个逻辑也蛮好玩的，有点破罐子破摔那味儿了。但我认为本质上是不同的，破罐破摔最后只能得到碎片，而如此思考却大概率能收获一良好的人际关系，美满的一段人生。大多数人之一生，不就是追求如此吗？
所以，孤独就孤独，随他去吧。

写点什么

发表于 2020-01-06 Valine：

写在开头

好久没有在博客上更新自己的一些想法了。之前有写过一些东西，但事后看来里面的观点太过于片面因此又把文章删去了。人是这样，代码也是这样，站在一个更高点俯瞰之前的自己或代码的时候总是觉得幼稚：哎，没想到我居然说出这样的话写过这样烂的代码。
当然了，今天不是来吐槽自己的。想写这篇博客的原因是因为这阵子想了不少东西，这阵子是那阵子呢，大概是从大一下暑假开始到现在吧；想什么呢，想过去，想未来，想人生，想理想。

想想过去

两个关键字，后悔、总结。
我自认是一个有远见的人，一直以来皆是如此定位我自己。但我更是一个自负的人，自负到我以为我自己是一个有远见的人。想想看，我若是足够有远见，大一就应该花他两周学完C参加学校的ACM新生赛然后以特长生的身份进入信息学院的计算机类专业而不是被动地进入电子类专业；我若是足够有远见，我就应该在大一多学点专业知识，以让现在的自己的根基更为牢固；我若是足够有远见，我就应该在大一下抓住那因自己的愚钝已经错失过一次的机会直接转入计算机系而不是留在通信工程；我若是足够有远见，我就不应该在大一下仍想着要继续留任自己的部门，险些做出现在可能让我更为后悔的事……
我意识到我终究是一个执行力低下，目光短浅的普通人罢了。
所以我总结，我想把“是”变为“曾是”。我总结，自己目光短浅是因为自己收集信息的能力还是不够，没有用心没有尽力去理解所处环境的游戏规则；我总结，自己执行力低下是因为自己迷茫，自己陷入虚无、无意义的泥沼中无法自拔。
那我现在变得更好了吗？可以说算是吧。至少不是那么迷茫了，至少知道大概如何去获得自己想要的东西了，至少更为成熟了。这段时间来，因为CTF的缘故，我能明显地感觉到自己对自己专业知识前所未有的把控力，自己的知识广度得到了前所未有的提升；因为大量阅读的缘故，自己的眼界变得更为开阔，思维也更为开放，心态也更为平和。可以说，参加CTF、大量的阅读是我这大学一年半做得最为正确的决定。
过去的迷茫已经过去了，也让他过去吧。未来才是我想要的，才是我大展拳脚的舞台。

想想未来

这阵子对未来的一些想法明晰了很多。从前我总是有这样一种恐惧：过那种一眼可以望穿的人生。这种恐惧束缚住了我，让我焦虑让我迷茫。如今想想，那个时候看了几本书就以为自己看破红尘的自己实在是有些搞笑。人生、生活确实是毫无意义的啊，这点确实如是，也不必欺骗自己。我从前总是在想我活着，我存在的意义何在，如果我终究只是那一抔黄土我又为何存在？我拥抱存在主义的思想，我希望他能给予我一些心灵的慰藉。而事实上，空虚感依旧，无意义感依旧，迷茫感依旧。但生命本身就是毫无意义的啊，我再怎么费心尽力去寻找那一点点可怜的意义感也是找不到的啊。我反思，我顿悟，既然如此，那不如直面虚无。无意义又如何，虚无又如何？古往今来，总是思想的解放带来生产力的解放，百家争鸣、文艺复兴大都如此。宏观上世界如是，微观上个人如是。思想的枷锁被突破了，随之而来的便是行动的身轻自如。我想明白这些后，随之而来的就是一片澄澈了。已经无任何意义了，与其苦苦追寻不如顺其自然让自己开心的过完这无意义的人生吧。
我以为，寻找一种意义找的并不是意义本身，而是一种让自己活下去的依靠。人总是需要些念想的。我妈妈说，她的念想就是我们两兄弟和她的老公，这是她活下去的意义。但念想意义终究是他物，人生一世，还是为自己而活的。依赖他物而活，多少有些将自己交出去的意思，有点不想自己对自己负责任的意思，有点逃避的意思。
是啊，寻找意义不过是种逃避罢了。我的恐惧也不是对人生的一眼望穿的恐惧，而是对生活中种种挑战的恐惧，我不过是在逃避罢了。那好的，我不逃了。
哎，太久没写东西，写得东西多少有点意识流的味道。太散了。总而言之，未来我不想在借意义之口逃生活之难了。

写在后头

其实也啥好写的，就是想到啥写啥吧。算是留下自己的一些东西吧。

总的来说，正如我前面所讲的，学会SQL语句并不是一件很难的事，若是能够集中精力，3天学完不成问题。但显然，学完这本基础教程对于开发一个高效的系统肯定是远远不够的，因为这本书也的确只是包含了SQL语句讲解这样十分基础的内容，像数据库原理这些比较底层的内容但却与高性能系统的开发息息相关的内容却鲜有提及或者说几乎没有。所以我的总结也只是对本书说提及的SQL语句进行的总结。当然了，这也只是一个我个人的总结，对于我这个刚学会SQL语句、没有任何开发经验的小白而言，也绝不会有什么开发的经验技巧在本篇博客中提及。如果你在阅读本篇博客时发现有什么描述不当或者错误的地方，请在评论区或者通过发邮件告诉我。
那么，开始做这个总结吧。

具体总结部分

本篇博客将从三个方面对这本书进行总结，数据库与SQL关系、基本数据库操作以及进阶数据库操作。

一、数据库与SQL关系

1.关于数据库

数据库（database）是将大量数据保存起来，通过计算机加工而成的可以进行高效访问的数据集合。而用来管理数据库的计算机系统则被称为数据库管理系统（Database数据库（database）是将大量数据保存起来，通过计算机加工而成的可以进行高效访问的数据集合。而用来管理数据库的计算机系统则被称为数据库管理系统（Database Manager System，DBMS）。数据库拥有多种，目前被广泛使用的数据库为关系型数据库（Relational Database），而所谓的用于管理数据库的SQL语言，就是用来对关系型数据库进行操作的。

2.关于SQL语句。

国际标准化组织（ISO）为SQL语句制定了相应的标准，以此为基准的SQL语句被称为标准SQL。事实上，不同的RDBMS使用的少量的SQL语句会有些许不同，但大同小异，本篇总结所提及的SQL语句都将是标准SQL语句。SQL语句可被分为三类，DDL、DML以及DCL，其中中间的D、M、C分别为英文Defination、Manipulation、Control。在书写规则上，SQL语句要以“;”结尾，SQL的关键字也不区分大小写（但数据库名区分大小写）。

3.两者之间的关系

关系就是SQL是用来操作关系型数据库（数据库中的一种）的语句。这个关系其实很容易就理解了。

比较运算符无非就是“>”、“<”这样的大家都懂的东西，不提。但需要去注意的是SQL语句中的不等号为“<>”以及不能对NULL使用比较运算符，除此之外别无其他值得注意的地方;算数运算符也无非是加减乘除这样的很基本的四则运算，也不多讲，需要注意的是，所有运算只要操作数中包含了NULL则最后的结果都为NULL；而逻辑运算符也无非是大家所熟悉的与或非，不提。主要值得去注意的是字句、关键字和聚合函数。

其实子句也挺简单的，也就是FROM，WHERE，GROUP BY,HAVING,ORDER BY这几个，这几个在SELECT语句中的顺序就像前面所列的那样，对这几个子句的具体功能做点解释:
FROM子句用于确定所要进行操作的表格；
WHERE子句用来执行查询的条件，也可以用来确定其他动作的条件，这个很好理解；
GROUP BY语句用于对表格进行汇总处理的情况，比如说记录相同特性物件的总数时会使用到GROUP BY子句；
HAVING子句用于对聚合结果进行处理的情况，至于聚合结果指的是经过聚合函数处理过后得到的数据；
ORDER BY子句则是对最后的结果进行排序时使用的子句。
这里可以记一下各个子句之间在计算机内部的具体的执行顺序

关键字平时会经常用到的是AS和DISTINCT，前者用于别名的安排，后者用于一列里重复项的消除。

而所谓的聚合函数呢，主要用到的也就那几个，分别是COUNT(),SUM(),AVG()。
COUNT()用于计算表格的行数，参数列表中填写数据名或者直接上“*”;
SUM()用于求和；
AVG()用于求平均数。

首先要来讲讲的是什么是视图以及视图和表的一个关系。书中是这样来说的：“从SQL的视角来看视图就是一张表”。而按照我的理解的话，本质上来讲，视图就是一条被保存了的SELECT语句。当你使用视图的时候，视图会通过这条SELECT语句创造出一张临时的表。那么视图和表是一个什么关系呢？按照上面的叙述来看，我认为这两者之间的关系是很明显的了。视图应当是依存于现有的表存在的，当你创建了一个视图后，表在视图在，表灭视图灭。但视图的存在与否与表却没有多大的关系。
视图最重要的有点是可以在大项目里节省空间，这边也就不再赘述了，直接来看怎样使用视图吧。我们通过这样的方式来创建一个视图：

可以对视图进行同表一样的操作。除此之外还有一件比较有趣的事，可以以视图为基础再创建一个新的视图，这个新建的视图叫多重视图。但需要注意的是，在执行多重视图的时候回降低数据库的性能，所以要尽量少用。
视图有两个限制：
1、对视图不可以使用ORDER BY子句；
2、对视图进行更新时需要满足以下条件：
A.创建视图的SELECT子句时没有使用DISTINCT；
B.FROM子句中只有一张表；
C.未使用GROUP BY子句；
D.未使用HAVING子句。
我们一开始提到视图与表的关系时有提到说视图依存于表存在，但事实上视图也是可以影响到表的，比如进行更新时就会影响到原表：你对视图进行了更新，表也会产生相应的更新；而你对表更新了，你会发现视图也跟着更新了。

SLIP,Compressed SLIP and PPP

In my opinion, the relationship between these three technologies is that the later one is the former ones upgraded version. And I am gonna summarize the three technologies one by one.

1.SLIP

We need to know that, the SLIP, CSLIP, and PPP are all the forms of encapsulation on the serial line which is in the Linke Layer.
About the SLIP, you can find more details in the RFC 1055. As a result, I will just mention some shortcomings of this tech in this blog. The format of this Encapsulation is below:

The SLIP has three shortcomings that are worth noting:

  1. When delivering data, each end must know the other’s IP address;
  2. No type field in the frame. Because of this, It can’t be used for some other protocol at the same time when the serial line is used for SLIP;
  3. No chechksum(This term is something I didn’t mention in the Ethernet and IEEE 802 Encapsulation part. It is related to the CRC field at the end of the frame and It is a cyclic redundancy check that detects errors in the rest of the fame.).That means it is the higher layer’s responsibility to detect the datagram’s corruption. Which is also means, it is essential that the higher layer must provide some form of CRC.BTW, there is always a checksum for IP header, TCP header, and TCP data while the checksum is optional in UDP header and UDP data.
    2.Compressed SLIP
    Since SLIP lines are often slow, the scientist develops a newer SLIP called CSLIP which stands for the Compressed SLIP. The CSLIP normally reduces the 40-bytes header to 3 or 5 bytes and these smaller headers greatly improve the interactive response time.
    Faster and more effective, that’s why I call it the upgraded version of SLIP.
    3.Point-to-Point Protocol
    The PPP corrects all the deficiencies in SLIP and you can read the RFC 1548 and RFC 1332 to find more details about this tech. The format of the PPP frame is below:

    One thing I think we should note is the flag field in the PPP frame. The flag is in the begin and the end of the frame and the value of this flag is 0x7e. The flag value can not be changed, so the PPP needs to ESCAPE this byte when it appears in the information field.the technology to escape this byte called bit stuffing.
    Compared to the SLIP, the PPP has the following advantages:
  4. Support for multiple protocols on a single serial line;
  5. Cyclic redundancy check on every frame;
  6. Dynamic negotiation of IP addresses for each end;
  7. TCP and IP header are compressed similar to CSLIP;
  8. A link control protocol for negotiating many data-link options.

There is a limit on the size of the frame for both Ethernet and IEEE 802 Encapsulation. This characteristic of the Link Layer is called the MTU, it is the Maximum Transmission Unit.
